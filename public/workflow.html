v3 Workflow

Two distinct workflows: Day 0 sets up the project once with /scaffold. Day 1 delivers features repeatedly with EPAV (/evaluate → /plan → /apply → /validate).

What is v3

v3 replaces static boilerplate templates with AI-generated scaffolds driven by your architecture document and Figma design. Any stack, any project — one workflow.

WorkflowWhenSkillOutput
Day 0 Once, at project start /scaffold Complete scaffold — builds and runs with zero credentials
Day 1 Every feature, every sprint /evaluate → /plan → /apply → /validate Shipped feature with real auth, real DB, real integrations
Day 0 is never run via EPAV. /scaffold is its own skill. EPAV is only for Day 1 feature tasks from the dev tasks CSV.

Artifact Chain

These documents must exist before Day 0 starts. They are produced once by the product and architecture team.

BRD → PRD → Architecture Doc + ADRs → Dev Tasks CSV
ArtifactWhat it contains
BRDBusiness requirements
PRDFeatures and acceptance criteria
Architecture Doc + ADRsStack, data model, auth, security, NFRs, error format, and the reasoning behind each decision
Dev Tasks CSVEvery feature broken into tasks. Each row has a task ID, description, and definition of done (acceptance criteria). This is the input for every Day 1 EPAV cycle.

Setup

Add the MCP server

Add to .mcp.json in your project (Claude Code) or claude_desktop_config.json (Claude Desktop / Cursor):

{
  "mcpServers": {
    "nexus": {
      "command": "uvx",
      "args": ["--refresh", "--from", "nexus-toolkit", "nexus-mcp"]
    }
  }
}

Install the CLI

# macOS / Linux — one-liner
$ curl -fsSL https://nexus.coderstudio.co/install.sh | sh

Day 0 — /scaffold

Day 0

One-time project setup. Generates a complete production-grade scaffold from the architecture document and Figma export. Output: infrastructure + design system + UI shell + AGENTS.md + knowledge/.

Core rule: Day 0 is structure and standards only — no live integrations. Mock auth. Mock data. Zero external dependencies. npm install && npm run dev must work with no credentials configured.

Step 1 — nexus init

Run once in the terminal before opening the project in Claude Code.

# From anywhere — point at your new project directory
$ uvx --from nexus-toolkit nexus init /path/to/my-project

# Or from inside the project
$ cd my-project && uvx --from nexus-toolkit nexus init .

This drops into the project:

.claude/commands/scaffold.mdDay 0 skill — invoked by /scaffold
.claude/commands/evaluate.mdDay 1 EPAV skill
.claude/commands/plan.mdDay 1 EPAV skill

.claude/commands/validate.mdDay 1 EPAV skill
.claude/commands/epav.mdFull-cycle orchestrator
.claude/settings.jsonPostToolUse hook — auto-runs graphify update after every file edit

Then ingest your inputs before running /scaffold:

# Ingest the architecture document
# Use the MCP tool: ingest_architecture_doc
# → extracts arch sections to knowledge/rules/arch-summary.md

# Ingest the Figma export
# Use the MCP tool: ingest_figma_zip
# → extracts Figma components into the project
Open the project in Claude Code after nexus init. The MCP tools and /scaffold skill are now available in your session.

EVALUATE

Type /scaffold in Claude Code. The AI reads the arch doc + Figma and produces a 10-point summary before writing any code.

#AreaWhat is covered
1StackDecisions and why they were made
2Data modelTables, relationships, indexes
3InfrastructureWhat is needed from day one
4Auth flowMiddleware stack, route protection
5Error handlingResponse format, error strategy
6SecurityRules and constraints
7Design tokensColors (converted from Figma oklch/hex), typography, spacing, radii
8Component inventoryEvery component, variants, all states
9Layout patternsGrid, breakpoints, responsive behavior
10IconographyLibrary, sizes, style
Stops here. AI waits for your approval before proceeding to PLAN. Review the summary — confirm the stack, data model, and design tokens are correct.

PLAN

After approval, the AI produces a blueprint — no code yet. Four outputs:

1
Infrastructure boilerplate
Directory structure, database schema, environment variables, auth flow, error handler
2
UI shell
Component list with prop interfaces, page layouts, build order (dependency chain)
3
AGENTS.md
Project rules for every AI tool: stack summary, coding standards, security rules, git conventions, quality gates
4
knowledge/ structure
Coding standards, prompt templates, design system spec, implement-and-test patterns
Stops here. AI waits for your /apply approval before writing any files.

APPLY

Plan approved — the AI generates everything in one pass.

First step — always: resolve_package_versions MCP tool runs before any file is written. It installs packages in a temp directory and returns exact pinned versions from the real lock file. Never from AI memory.

What gets written:

CategoryFiles
Config.gitignore, .env.example, linter + formatter config
DatabaseSchema migration (SQL ready, not applied yet), seed script
AuthMock auth middleware — reads mock-role cookie, redirects unauthenticated requests
APIHealth-check endpoint only. No business logic.
Git hookshusky + lint-staged + commitlint (npm); equivalent for other stacks. chmod +x .husky/pre-commit runs automatically. CI sets HUSKY=0.
Design systemTailwind config, globals.css with converted hsl() tokens, font loading
ComponentsAll components from Figma with every visual state (default, hover, focus, disabled, loading, empty, error)
PagesAll pages from Figma with mock data — no API calls
Project rulesAGENTS.md at project root, knowledge/ with coding standards + design system spec
CIGitHub Actions: lint, typecheck, build
DocsREADME with local setup instructions

Mock auth pattern

Day 0 uses cookie-based mock auth — readable server-side, no real provider needed.

# Login page sets the cookie — no credentials checked
# Cookie: mock-role=admin (or manager, employee, etc.)
# Expiry: 8 hours

# Route guard reads the cookie and redirects if missing
# Server components call getMockSession() — no Context provider needed

# Real auth provider replaces this on Day 1

Package versions

# What the AI does — always
resolve_package_versions({
  packages: ["next@^16", "react@^19", "@supabase/supabase-js@^2"],
  stack_hint: "Next.js 16 TypeScript"
})
# → { "next": "16.2.9", "react": "19.2.7", ... }

# Then writes package.json with exact versions — no ^ or ~
# Then runs: npm install → package-lock.json
NEVER write "latest". NEVER write semver ranges. NEVER use versions from AI training data — they are always stale.

Do NOT implement in Day 0

WhatWhen
Real auth provider (Supabase Auth, NextAuth, OAuth)Day 1
Live database queries or mutationsDay 1
External service calls (email, storage, payment)Day 1
API endpoints beyond health checkDay 1
Dev tasks from the CSVDay 1

VALIDATE

Build must pass before anything else is checked.

StackBuild command
Next.js / Vitenpm run build
Flutterflutter build apk --debug
Gogo build ./...
Rustcargo build
Java / Springmvn package -DskipTests

19-point checklist after build passes:

GroupChecks
Packages (4)Stable versions, no deprecated APIs, lock file pinned, no CVEs
Infrastructure (6)Structure matches arch, schema matches data model, env vars documented, production logging, security headers, DB pooling
UI fidelity (6)Matches Figma, responsive at 320/768/1024/1440px, all visual states, WCAG AA, props typed for Day 1 wiring, no API calls in components
AGENTS.md + knowledge (2)Coding standards match arch, design system spec saved
Git hygiene (1)Pre-commit hooks configured and executable
Every [BLOCKER] and [FIX NOW] is fixed before Day 0 is called complete. npm install && npm run dev works. Zero credentials needed.

Day 1 — EPAV

Day 1

Runs for every feature task from the dev tasks CSV. Repeats for every sprint, forever. This is where real auth, real DB, and real integrations are wired.

Dev Tasks CSV row
↓
/evaluate → blast radius + acceptance criteria
/plan → implementation blueprint (no code)
/apply → write the code
/validate → all criteria pass, fix all blockers
↓
Ship the feature

/evaluate <task>

Input: one row from the dev tasks CSV.

# Example
/evaluate TASK-042 — Leave approval flow
  AC1: Manager can approve with optional note
  AC2: leave_balances.used_days increments atomically
  AC3: audit_log entry created on every status change
1
graphify blast radius
graphify query finds all files the task touches and what references them
2
knowledge check
Checks knowledge/ for relevant rules, patterns, and prior retros
3
Output
Affected files, acceptance criteria restated, risks flagged. Waits for approval.

/plan

1
graphify path check
graphify path A B confirms relationships between primary files
2
Blueprint
Numbered steps: file by file, what changes and why. Files created / modified / deleted. Which AGENTS.md rules apply.
No code written. Waits for /apply approval.

/apply

Implements exactly what the plan says — nothing more.

What gets wiredHow
Real authReplace mock cookie with real provider (@supabase/ssr, magic link, OAuth)
DatabaseApply migration (supabase db push), replace mock data imports with real client calls
Route HandlersWritten with Zod input validation, consistent error format from arch doc
RLS policiesUsing get_my_role() — never recursive subqueries
Multi-step mutationsAtomic transactions — all succeed or all roll back

graphify updates automatically via PostToolUse hook after every file written — no manual step.

/validate

1
Build must pass first
A failed build is an automatic [BLOCKER]. Nothing else is checked until it exits 0.
2
Acceptance criteria
Every criterion: [PASS] / [FAIL] / [PARTIAL]
3
Fix all blockers
Every [BLOCKER] and [FIX NOW] is fixed before the task is marked done
4
Contribute patterns back
New patterns discovered go to knowledge/patterns/ for future tasks

Key Rules

RuleDetail
graphify firstRun graphify query before reading any source file. The graph orients you — raw file reads come after.
No version guessingresolve_package_versions before every APPLY. Never write versions from AI training data.
No scaffold CLIsNever run npx create-next-app or equivalent. Write all files directly.
Figma colors → hsl()Figma exports oklch or hex. Convert to hsl() (web), Color() (Flutter), hex (RN) before writing design files.
Day 0 = mock onlyNo real auth, no real DB, no external calls. The project must build with zero credentials.
Stop before each phaseEVALUATE stops before PLAN. PLAN stops before APPLY. User approves each transition.
Fix all blockersEvery [BLOCKER] and [FIX NOW] is fixed before the phase is called complete. Never defer a blocker.
